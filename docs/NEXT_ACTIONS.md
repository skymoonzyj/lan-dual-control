## Mac Codex 可接任务

- 每次恢复工作、准备向 Windows 端发状态或准备正式密码验收前，先跑 `node scripts/mac/check-mac-resume-status.mjs --checkBoard` 做轻量总览：它只读检查当前 git/联络板/Mac host `/discovery`/权限/LAN 地址/旧 build runtime 源码差异，并给出下一步建议，不启动服务、不认证、不要求密码、不发送输入事件。需要直接发通讯板时用 `--boardSummary` 输出一段秘密安全摘要，避免手工拼错 host、权限、buildDiff 或下一步；摘要会明确提醒 inject 启动必须 `--confirmUserWatching`。需要脚本消费时加 `--json`；需要在自动化里把离线 host、工作区未清理或运行源码变化变成失败时分别加 `--requireOnline`、`--requireClean`、`--requireNoRuntimeChanges`。如果报告 Mac host 离线，正式端到端验收前用 `node scripts/mac/start-mac-host.mjs --promptPassword --requirePassword` 安全启动；如果只是 build 元数据落后且 `changedHostRuntimeFileCount=0`，通常可先继续 discovery/runtime 协调，部署强验收再重启。
- 准备正式呼叫 Windows 做端到端验收前，优先跑 `node scripts/mac/check-mac-formal-e2e-status.mjs --boardSummary`：它会默认读取联络板，并把 repo、Mac host、LAN 地址、`inputMode=log`、权限、H.264、系统 PCM、剪贴板、显示器和 buildDiff 汇总成正式 checklist；只有没有 blocker 且联络板可读时才会标记 `readyToCall=true`。如果需要机器可读状态，用 `--json` 读取 `checklist`、`counts`、`readyToCall`、`callText` 和 `callPayload`；确认要由脚本代发 Agent Link Board 呼叫时，再显式加 `--sendCall`，它只在 `readyToCall=true` 时发送一条无密 formal E2E call。未 ready 时会拒绝发送并失败，JSON 会保留具体 blockers、下一步和变动 runtime 文件；通讯板已有 `currentCall` 时默认拒绝覆盖。只有已经在联络板上协调清楚要替换旧呼叫时，才显式加 `--forceCall`。如果历史 formal call 仍挂在板上、但当前 checklist 已不 ready，可显式运行 `node scripts/mac/check-mac-formal-e2e-status.mjs --clearStaleCall --json`：它只会清理 Mac Codex 发起、Windows Codex 配合、目标为“正式端到端验收 Mac host”的过期 call；其他端或其他目标的 call 会保留。开发中本地预览可加 `--allowDirty`，正式发 call 前不要放宽 dirty worktree。该脚本不启动服务、不认证、不要求密码、不发送输入；`inject` 会明确跳过，仍需用户另行明确确认。当前真实 Mac host `192.168.31.122:43770` 已在 runtime build `c5e5009` 完成完整 Windows formal Mac E2E；后续 Mac 侧再发正式呼叫前，应先重新确认 host 是否仍在线和 build 是否当前，而不是复用旧 call。
- 如果 Supervisor 或 Windows 审查机复跑 Mac 侧测试，优先确认 `node scripts/mac/test-mac-password-prompt.mjs` 和 `node scripts/mac/test-mac-host-start-helper.mjs`：前者应使用 fake `osascript`/`swift` 的 Unix shim + `.mjs` + Windows `.cmd` 包装器，在 Windows 上优先直连 fake `.mjs` 执行体以避开 `cmd.exe` 多行 AppleScript 参数转义，不打开真实系统弹窗；后者在非 macOS 上应继续跑 dry-run/status/密码安全检查，并对真实 Swift Mac host 启动段输出清晰 `[SKIP]`。当前 Windows 审查机复跑这两条已通过；若在 Codex 沙盒里看到 `spawnSync node.exe EPERM`，用允许子进程的执行方式复跑确认。

- 日常启动真实 Mac host 前可先用 `node scripts/mac/start-mac-host.mjs --status` 只读查看当前 `/discovery`、runtime、权限、能力和 Windows 可连地址；它不会启动服务，也不会要求或打印密码。若运行中 build 落后当前 git，`--status` 会只比较 `apps/mac-host` 运行源码并提示旧 build 后是否有 host 行为相关改动，帮助判断是必须重启，还是只是 build 元数据落后；脚本或联络板自动化需要稳定字段时可加 `--json`，读取 `online`、`runtime`、`permissions`、`capabilities`、`lanAddresses` 和 `buildDiff`。需要启动时优先用 `node scripts/mac/start-mac-host.mjs --promptPassword --requirePassword`；该入口会先播放两声提示音，并输出一行不含密码的前台弹窗提示，再优先打开 macOS 系统隐藏密码弹窗，系统弹窗打不开时才尝试原生 AppKit 前台高层级隐藏密码框，默认 `inputMode=log`、打印 Windows 可连接的局域网地址、等待 `/discovery`，并跑只读 runtime/display 校验。显式 `--promptPassword` 会重新弹窗输入，不会静默复用已有 `LAN_DUAL_PASSWORD`。如果希望启动助手完成 `/discovery` 和 runtime/display 校验后退出、让 host 继续常驻，可显式加 `--background`，日志默认在 `.dev-lab/mac-host/`，也可加 `--logFile <path>`；后台模式仍只有在校验通过后才脱离，且子校验不通过 argv 传密码。需要停止后台 host 时用 `node scripts/mac/start-mac-host.mjs --stop` 或 `--stop --json`，它只停本机 `/discovery` 对应的 macOS host PID，不读取密码，也会拒绝非本机或非 Mac discovery。需要先恢复 `/discovery`、runtime/build 和权限诊断但暂时不能共享正式密码时，可用 `--ephemeralPassword --requirePassword` 生成一次性随机密码且不打印；这只适合无密码 discovery/runtime 检查，认证联调仍需用户输入正式密码或重启到约定密码。直接 `swift run lan-dual-mac-host` 时源码默认也应保持 `log`，`check-mac-host-readiness` 默认会回归该安全默认值；日常真实注入一律走启动助手，必须有人在屏幕前确认安全后再显式加 `--inputMode inject --confirmUserWatching` 或 `--injectInput --confirmUserWatching`。裸 `LAN_DUAL_INPUT_MODE=inject` 只能作为人工监督下的底层调试方式，不作为交接命令。
