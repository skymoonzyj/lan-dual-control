- Windows host 一键反控请求策略相关改动后，先跑 `node scripts/windows/test-windows-host-reverse-control.mjs`：它覆盖未认证 `LAN002`、默认 `deny` 的 `LAN008` 安全拒绝和最近请求状态、一次性临时授权消耗、缺少 `requestId`、显式 `accept` 实验策略和 `disabled` 能力声明；命令行授权助手相关改动再跑 `node scripts/windows/test-windows-reverse-control-grant-helper.mjs`，确认 `--status/--grant/--revoke`、`--json`、`--boardSummary` 和离线失败摘要都不泄密、不发送输入、不执行 `inject`。当前默认策略仍是安全拒绝，真正产品化同意需要后续接桌面确认入口。
- Windows host 或 Mac client 视频参数相关改动后优先运行 `node scripts/windows/observe-windows-host-media.mjs --resourceSampleTree true --json`，顺序确认普通启动下 60 Hz 视频请求、实际 FPS、最大帧间隔、采集管线、请求码率、`jpegQuality`、帧新鲜度、WASAPI 音频帧、资源摘要和 `summary.status`；需要同步到通讯板时改用 `--boardSummary` 输出一行无密媒体基线，单路成功单路失败会标为 `Windows media: partial`，全部执行链路失败才标为 `Windows media: failed`，便于直接说明 FFmpeg/GDI/WASAPI 卡在哪里。视频或音频其中一路失败时，脚本仍会继续尝试另一条链路并保留成功结果，避免一次抓屏波动把音频基线也丢掉。普通输出模式下视频、音频和聚合观察都会默认每 10 秒打印进度；现场长测可加 `--progressIntervalMs 5000`，自动化 JSON / `--boardSummary` 会保持干净不混入心跳。若只看视频，再运行 `node scripts/windows/observe-windows-host-video.mjs --fps 60 --useDefaultMaxScreenFps --expectSessionFps 60 --durationMs 4000 --minFrames 140 --minFps 35 --requireMonotonicTimestamp --maxFrameAgeMs 1000 --resourceSampleTree true`。可再用 `--bandwidthKbps 5000 --qualityPreset smooth --json` 与 `--bandwidthKbps 40000 --qualityPreset sharp --json` 对照低/高码率；最后运行 `node scripts/windows/test-mac-client-browser.mjs` 确认 Mac client 默认 1080P/60Hz/20Mbps 和 2K/60Hz/40Mbps 更新路径未退化。
- 继续维护本机假 Mac 服务，用于快速回归和失败场景模拟。
